# Product Document Contract

New products use the guided Founding Hypothesis. Follow-up features use the guided Peter Yang PRD without Meeting Notes. Template guidance, arbitrary sections, and direct user edits remain valid; guidance is never accepted product content.

Discussion does not write. Explicit creation returns the unanswered template. Explicit edits change only named content. Agent suggestions remain proposed until acknowledged. Never infer non-goals, security boundaries, rollout, maturity gates, metrics, or acceptance.

Promotion allocates a stable Epic ID and filename without changing document bytes. Acceptance requires an answered observable `Acceptance`, `Done when`, or legacy `Product Acceptance` section and binds the exact source digest. Later edits require explicit re-acceptance.

Legacy comprehensive PRDs remain readable and launchable. Their existing Scope Areas and machine fields stay valid; new flexible documents receive a deterministic default Epic Scope Area in compiled controller state rather than product prose.
